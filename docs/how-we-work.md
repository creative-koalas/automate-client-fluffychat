# PsyGo 工作方式与过程管理

## 一、核心思想

### 公司是一台运行中的机器，不是一张组织架构图

公司的本质不是 structure（组织结构），而是 **operations**（运营过程）。

- **Structure 只是 runtime**。人也好，AI Agent 也罢，都是执行任务的运行时环境。谁能跑这个任务，谁就去跑。能力匹配，不看身份。
- **存储是 GitHub Issues**。一切工作的定义、状态、历史，都记录在 Issue 里。不在 Issue 里的工作，等于不存在。
- **过程即产品**。我们不开无结论的会，不写无人看的文档。每一个动作都对应一个 Issue 的状态变更。

这意味着：
1. 如果你是新人，你不需要去理解"公司的汇报关系是什么"，你只需要知道：**你的任务在哪里，状态是什么，什么时候要完成**。
2. 如果一个 AI Agent 能干的活，就不需要人来干。人应该去做 AI 做不了的事情。
3. 团队扩缩容 = 增减 runtime。加一个人和加一个 Agent，在过程管理上是同构的。

## 二、工作流：一切围绕 Issue

### Issue 是工作的原子单位

每一件要做的事，都是一个 Issue。Issue 包含：

- **标题**：一句话说清楚要做什么
- **描述**：背景、预期结果、验收标准
- **标签**：类型 + 难度（见下文）
- **评分**：4 维度量化优先级
- **负责人**：谁来执行
- **排期**：什么时候开始、什么时候结束

### 标签规范

每个 Issue 恰好 **1 个类型标签 + 1 个难度标签**。

**类型（5 个）：**

| 标签 | 含义 |
|---|---|
| `bug` | 功能异常或错误 |
| `feature` | 新功能开发 |
| `ux` | 用户体验优化 |
| `task` | 非开发类任务（测试、申请等） |
| `ops` | 运维 / 平台相关 |

**难度（4 个）：**

| 标签 | 含义 | 预估用时 |
|---|---|---|
| `easy` | 改动范围小，逻辑清晰 | 半天 |
| `medium` | 涉及多模块或前后端联动 | 1 天 |
| `hard` | 跨平台或架构级改动 | 2-3 天 |
| `complex` | 大型功能，多平台 + 前后端 + 第三方集成 | 3-5 天 |

不使用优先级标签——优先级由量化评分决定，不靠人拍脑袋。

## 三、优先级：量化评分，不靠感觉

### 4 个评分维度（1-5 分）

| 维度 | 说明 | 5 分 = |
|---|---|---|
| 用户影响 | 对用户体验的影响程度 | 核心功能不可用 |
| 紧急程度 | 是否需要立即处理 | 阻塞发布 |
| ROI | 投入产出比 | 小投入大收益 |
| 技术复杂度 | 实现难度 | 非常复杂（此项为负向指标） |

### 综合得分公式

```
综合得分 = round((用户影响 + 紧急程度 + ROI - 技术复杂度) × 10 / 13)
```

满分 10 分，越高越优先。这个公式的逻辑是：**价值高、紧急、性价比好、实现简单的事情先做**。

综合得分决定 Issue 的排列顺序，而不是某个人说"这个重要"。

## 四、人员分配：技能匹配，不看职级

分配原则：

1. **能力匹配优先**。根据每个人的技能方向分配任务，而不是按职级或工龄。
2. **保密边界**。核心模块（如记忆模块）有保密要求，实习生无法参与，由全职人员处理。
3. **兜底机制**。有人专门承接"分不出去的任务"，确保没有任务被遗漏。
4. **协作模式**。大型功能（如 `complex` 难度）由多人协作：一人主导前端，一人负责后端。

---

## 五、排期：紧凑排列，不留空档

### 规则

- 任务之间**无空档**（周日和法定假期除外）。
- 上一个任务半天结束，下午立即接下一个任务。开始/结束日期可以是同一天。
- **分批执行**：按优先级分为第一批 ~ 第四批 + 暂缓，每批内部并行推进。

### 排期三处同步

任何排期变更，必须同步更新三个地方：

1. 本地排期 Markdown 文件
2. 导出的 PDF
3. GitHub Project 中的字段（用时 / 开始 / 结束）

三者不一致 = 排期无效。

### Issue 关闭后的处理

关闭一个 Issue 后：
- 从 Project 归档
- 该负责人的后续任务日期**前移填补空档**
- 重新计算排期

## 六、工具链

| 工具 | 用途 | 角色 |
|---|---|---|
| **GitHub Issues** | 工作的存储层 | 一切任务的 single source of truth |
| **GitHub Project** | 规划看板 | 评分、排期、状态跟踪 |
| **gh CLI** | 命令行操作 GitHub | 批量操作、自动化脚本 |

### 为什么用 Issue 而不是用文档、群聊、会议

- **Issue 有状态**。Open / In Progress / Done，一目了然。
- **Issue 有历史**。谁创建的、谁评论的、什么时候关闭的，全部可追溯。
- **Issue 可量化**。评分、排期、负责人，都是结构化数据。
- **Issue 可自动化**。可以通过脚本和工具批量操作 Issue。

群聊里说的话会淹没。会议上达成的共识没人记录。只有 Issue 是持久的、可查询的、可操作的。

## 七、招聘：一天完成，实战说话

招聘流程设计也体现了同样的思想——**看实际产出，不看简历故事**。

1. **实战题当天发、当天交**。早上 9:00 发题，下午 17:00 前提交 GitHub 仓库。
2. **代码必须能跑起来**。跑不起来直接淘汰，没有解释空间。
3. **面试围绕候选人自己的代码**。不问八股文，不考算法题，共享屏幕讲你写的代码。
4. **鼓励用 AI 工具**。我们在乎的是你能不能理解和掌控你写的东西，而不是你是不是纯手打。
5. **整个流程一天完成**。不拖泥带水。

## 八、思维方式：给新人的几条建议

1. **先看 Issue，再写代码**。不要一上来就写，先确认 Issue 的描述、验收标准、评分维度。写完后对照检查。

2. **你的工作 = Issue 状态的变更**。每天的进展应该体现为 Issue 从 Todo → In Progress → Done。如果你忙了一天但没有任何 Issue 状态变化，要反思这一天做了什么。

3. **遇到问题，更新 Issue**。而不是在群里说一句"卡住了"。在 Issue 里评论你的进展、遇到的阻塞、需要的协助。这样任何人都能接手。

4. **不要等排期，看综合得分**。得分高的先做。排期是计划，得分是优先级。如果你提前完成了当前任务，直接看下一个得分最高的。

5. **人和 AI 是平等的 runtime**。不要觉得"这个活太简单了不想做"。如果简单到 AI 能做，那就应该交给 AI。你去做 AI 做不了的更有价值的事。

6. **代码提交 = 交付物**。提交的代码要能跑，要有合理的 commit 历史，要有清晰的变更说明。单次 commit 提交全部代码是不被接受的。

---

## 九、总结

```
公司 = Operations
人/AI = Runtime
Issue = Storage
评分 = 调度算法
排期 = 执行计划
```

我们不是一个"管人"的公司，我们是一个"管过程"的公司。人会变，AI 会进化，但过程管理的逻辑不变：**定义清楚要做什么、量化清楚先做什么、分配给能做的 runtime、跟踪到完成。**
